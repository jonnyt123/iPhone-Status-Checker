Build a production-ready web app called “iPhone Status Check Canada”.

Purpose:
A paid website where a customer enters their email first, then enters an IMEI or serial number, pays CAD $0.99, and receives the device-check results by email. The app must use an external provider integration patterned around imeiapi.org. The app must only show statuses that are actually returned by the configured provider. It must never guess, infer, scrape, or fabricate Apple-related statuses.

Core business rules:
- Charge exactly 0.99 CAD per order
- Require customer email before allowing device identifier entry
- Accept IMEI by default
- Accept serial number only if provider support is enabled by environment variable
- Run the provider lookup only after successful payment
- Email the results to the customer after payment succeeds
- If a field is not returned by the provider, show “Unavailable from provider”
- Add clear notices:
  - “We return results only from authorized data sources.”
  - “Blacklist coverage depends on the provider and region.”
  - “Some checks may be unavailable depending on provider support.”
  - “Apple-related statuses are shown only when returned by the configured provider and are not inferred.”

Stack:
- Next.js with TypeScript
- PostgreSQL with Prisma
- Stripe Checkout for one-time payment
- Stripe webhook handling
- SendGrid for transactional email
- Zod validation
- Rate limiting
- Basic admin dashboard
- Clean responsive mobile-first design
- Secure environment-variable configuration

Pages:
- Home page with explanation and CTA

- Admin page

User flow:
1. User lands on homepage
2. User clicks start
3. User enters email address
4. User enters IMEI or serial number
5. User sees price of CAD $0.99
6. User is sent to Stripe Checkout
7. Stripe payment completes
8. Stripe webhook marks order paid
9. App calls provider adapter
10. App stores normalized results and encrypted raw provider response
11. App emails results to user
12. User sees success screen telling them results were emailed

Provider behavior:
- Primary provider integration should be implemented as a configurable adapter based on imeiapi.org request/response flow
- Use environment variables for provider base URL and API key
- Build the provider layer so other providers can be added later
- Do not assume the provider always returns activation lock, find my, or blacklist data
- Only map fields that are actually present
- Missing fields must become “unavailable”
- Preserve source/provider name in the results
- Log provider errors cleanly
- Retry safe transient failures once or twice
- Never make unsupported claims about Apple-authoritative results
